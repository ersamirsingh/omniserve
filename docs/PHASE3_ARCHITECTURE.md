# Phase 3 Architecture Notes

This document highlights the design patterns, classifications, and technical details of the Phase 3 operational split implemented on the **FoodMesh Platform**.

---

## 1. Classification & Sourcing Strategy

Operational mode filtering is managed authoritatively by the backend. The frontend does not independently decide classification.

### ONLINE Classification
* **Filters**: `["SWIGGY", "ZOMATO", "WEBSITE", "ONLINE", "DELIVERY", "TAKEAWAY", "ONDC", "WHATSAPP"]`
* **Route**: `/operations/online`
* **Behavior**: Renders orders originating from aggregator webhooks and web storefronts.

### DINE_IN Classification
* **Filters**: `["DINE_IN", "QR_DINE_IN", "WAITER", "POS"]`
* **Route**: `/operations/dine-in` (with sub-tabs for floor layout, tables, reservations, and KDS channels)
* **Behavior**: Renders dine-in table QR, waiter console, and local cashier orders.

---

## 2. Shared Workspace Component Reuse

* **No Code Duplication**: `OrdersPage.jsx` has been refactored to accept a `mode` prop (`mode="ONLINE" | "DINE_IN" | "ALL"`).
* **Deep Link Compatibility**: `/orders` is preserved in `mode="ALL"`. This ensures existing query deep links (`/orders?orderId=xyz`) generated by notification engines or email alerts continue to resolve and load the details drawer without breaking.

---

## 3. Realtime Updates

* **Scoped Refetch**: Real-time WebSocket triggers (`ORDER_CREATED`, `ORDER_STATUS_CHANGED`) refresh the workspace lists by dispatching a scoped refetch:
  `GET /api/v1/orders?operationalMode=ONLINE` or `GET /api/v1/orders?operationalMode=DINE_IN`
* **KDS Agnosticism**: The Kitchen Display System remains source-agnostic, displaying prep tickets for both online and offline orders.

---

## 4. Technical Debt Note (Source/Fulfillment Overloading)

* **Limitation**: The persisted database `source` field in the `Order` model currently serves a dual purpose: storing both the provider channel origin (e.g. Swiggy, Website) and fallback fulfillment-derived values (e.g. Delivery, Takeaway).
* **Future Recommendation**: Run a database schema migration to split these fields into `Order.source` (the channel provider origin) and `Order.fulfillmentType` (the dining behavior) once a full deployment transition plan is scheduled.
